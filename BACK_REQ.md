## 🔴 Блокеры публичного релиза

### BR-2. Зеркальная проверка почтового домена в `POST /auth/register` (149-ФЗ)
**Что:** отклонять регистрацию с email не из списка российских почтовых доменов —
HTTP 400 с понятным `detail` (например, `"Registration requires a Russian email provider"`).
**Зачем:** 149-ФЗ (поправки от 01.12.2023) — регистрация только через российские
почты/номера/ЕСИА/российские ID. Клиентская проверка уже есть, но обходится
прямым HTTP-запросом — юридически прикрывает только серверная.
**Важно:** проверка ТОЛЬКО на регистрации. `/auth/login` и `/password-reset/*`
не ограничивать — существующие аккаунты с иностранной почтой должны работать.
**Android:** готово — `EmailValidator.isAllowedDomain` + `AllowedEmailDomainsRepository`,
список: Яндекс (`yandex.ru`, `ya.ru`, `yandex.com`, `narod.ru`),
VK/Mail.ru (`mail.ru`, `bk.ru`, `list.ru`, `inbox.ru`, `internet.ru`, `vk.com`),
Rambler (`rambler.ru`, `lenta.ru`, `autorambler.ru`, `myrambler.ru`, `ro.ru`).
Списки клиента и сервера должны совпадать.
**Статус бэка (ревизия 17.07):** ❌ не реализовано — в `register_user` и схеме
`UserCreate` проверки домена нет.
**Приёмка:** `curl POST /auth/register` с `user@gmail.com` → 400; с `user@yandex.ru` → 200.

### BR-3. Хеширование паролей — перейти на bcrypt
**Что:** сменить схему хеширования паролей с `sha256_crypt` на `bcrypt` (или argon2).
**Зачем:** быстрые хеши брутфорсятся на GPU; при утечке БД все пароли скомпрометированы.
**Статус бэка (ревизия 17.07):** ❌ подтверждено — `app/core/security.py:10`:
`CryptContext(schemes=["sha256_crypt"], ...)`. При этом `passlib[bcrypt]` уже
в `requirements.txt` — достаточно сменить схему + миграция хешей при логине
(`deprecated="auto"` в CryptContext перехеширует прозрачно:
`schemes=["bcrypt", "sha256_crypt"]`).
**Приёмка:** новые хеши в таблице `users` начинаются с `$2b$` (bcrypt) / `$argon2`;
логин под старым паролем перехеширует запись.

### BR-15. Публичный URL политики конфиденциальности
**Что:** опубликовать политику конфиденциальности по публичному URL
(например, `https://runtastic.gottland.ru/privacy`) — обычная HTML-страница,
доступная без авторизации.
**Зачем:** RuStore (и Google Play) требуют ссылку на политику в карточке
приложения — без неё публикация невозможна (поэтому статус поднят до 🔴).
Также 152-ФЗ требует опубликованного документа о политике обработки ПДн (ст. 18.1).
**Android:** черновик текста — в `LegalScreens.kt` (`PrivacyPolicyScreen`),
структурирован по 152-ФЗ. Перед публикацией: заполнить реквизиты оператора
(плейсхолдеры в тексте), юр-проверка. Тексты в приложении и по URL должны совпадать.
**Статус бэка (ревизия 17.07):** ❌ нет — `StaticFiles` смонтирован только на
`/static` (аватары/иконки), страницы политики нет.
**Приёмка:** URL открывается без авторизации, отдаёт актуальный текст политики;
ссылка принята кабинетом RuStore.

## 🟡 Высокий приоритет

### BR-16. Пульс (`heart_rate`) в GPS-точках
**Что:** принимать и хранить nullable-поле `heart_rate` (int, уд/мин) у каждой
точки в `POST /training/{training_id}/gps_points`; отдавать его в ответе
`GET /training/{id}/get_training` (параллельным массивом рядом с
`gps_points_timestamps` или полем в точках — формат согласовать).
Опциональное развитие: агрегаты `avg_heart_rate`/`max_heart_rate`
в `POST .../save_training` и в `GET /training/history`.
**Зачем:** Android подключает внешние BLE-пульсометры и пишет пульс в каждую
GPS-точку. Без серверного поля данные теряются: история тренировок не сможет
показать средний/максимальный пульс и график пульса.
**Android:** готово — клиент УЖЕ отправляет `heart_rate` в каждой точке
(`GpsPointDto`, null если датчик не подключён; Gson null-поля не сериализует,
поэтому старые клиенты и тренировки без датчика ничего не ломают).
**Статус бэка (ревизия 17.07):** ❌ подтверждено — в pydantic-схеме `GPSPoints`
поля нет, в Cassandra-insert (`app/services/cassandra.py:save_gps_points`)
колонки нет → значения молча теряются. Понадобится колонка в Cassandra-таблице
`gps_points` + прокидывание в `get_training`.
**Приёмка:** `POST .../gps_points` с `heart_rate: 150` сохраняет значение;
`GET .../get_training` возвращает его для точек трека; точки без поля/с null
принимаются как раньше.

### BR-4. `GET /auth/allowed-email-domains` — серверный список доменов
**Что:** новый публичный эндпоинт: `{"domains": ["yandex.ru", "mail.ru", ...]}`.
**Зачем:** сейчас список захардкожен в APK — любое изменение (новый провайдер,
корпоративные домены РФ, вузовские `@petrsu.ru`) требует релиза приложения.
**Android:** инфраструктура готова — заменить `AllowedEmailDomainsRepositoryImpl`
на сетевую реализацию с кэшем (план в TODO-комментарии внутри impl),
хардкод останется fallback'ом при недоступности сети.
**Статус бэка (ревизия 17.07):** ❌ эндпоинта нет в OpenAPI-схеме.
Логично делать вместе с BR-2 — один источник списка для проверки и выдачи.
**Приёмка:** эндпоинт отдаёт актуальный список; изменение списка на сервере
подхватывается клиентом без обновления APK.

## 🟢 Средний / низкий приоритет

### BR-9. Коды верификации через `secrets`, не `random`
`random` предсказуем — коды подбираются. `secrets.randbelow` / `secrets.choice`.
**Статус (17.07):** ❌ подтверждено — `app/core/security.py:42`:
`random.choices(string.digits, k=length)`. Замена — одна строка.

### BR-10. CORS — убрать `*` вместе с credentials
**Статус (17.07):** ❌ подтверждено — `app/main.py:50-54`:
`allow_origins=["*"]` + `allow_credentials=True` одновременно. Мобильному
клиенту CORS не нужен вовсе — можно сузить до списка доверенных origin'ов
или убрать credentials.

### BR-12. Timezone — единообразие UTC
**Статус (17.07):** ❌ подтверждено, локализовано — в `app/services/auth.py`
создание кодов использует aware `datetime.now(timezone.utc)`, а проверка —
naive `datetime.now()` (строки ~125 `expires_at > datetime.now()` и ~145
`verified_at = datetime.now()`). Пока TZ сервера = UTC, работает случайно;
на сервере с другой TZ коды «протухают» раньше/позже срока.

### BR-17 (новый). Лимит попыток верификации сбрасывается повторной отправкой
**Что:** `attempts` живёт на конкретном коде; `/auth/resend-code` создаёт новый
код с `attempts=0`. Атакующий может циклом «5 попыток → resend (кулдаун 120с) →
5 попыток» перебирать бесконечно, лимит `MAX_VERIFICATION_ATTEMPTS` не копится.
**Зачем:** остаточный риск после закрытия BR-8. При 6-значном коде и 10-минутном
TTL практическая опасность низкая (≈25 попыток на код), поэтому 🟢.
**Как:** считать суммарные попытки на пользователя за окно времени, либо
ограничить число resend'ов на одну регистрацию.

### ✅ BR-8. Инвалидация старых кодов верификации

не предъявляются — окно перебора на них не расширяется. Физически старые строки
остаются в таблице (мусор, не уязвимость). Остаточный риск со сбросом счётчика
попыток вынесен в BR-17.

латентность SMTP), но остальные запросы не страдают — суть задачи закрыта.
